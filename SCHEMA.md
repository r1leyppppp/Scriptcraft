# 剧本 YAML Schema 定义文档

## 概述

Scriptcraft 使用 YAML 作为剧本的结构化存储格式。本文档定义了剧本 YAML 的完整 Schema，包括每个字段的类型、是否必填、含义说明，以及该 Schema 的设计原因。

---

## Schema 结构

```yaml
scene:
  location: string      # 必填，场景地点，例如"青石岭祖祠后院"
  time: string          # 必填，时间描述，例如"深秋黄昏"
  atmosphere: string    # 选填，氛围描述，例如"肃穆中透着苍凉"

characters:
  - name: string        # 必填，人物姓名
    appearance: string  # 选填，外貌描述
    personality: string # 选填，性格特征

actions:
  - string              # 选填，场景动作描写，可有多条

dialogue:
  - speaker: string     # 必填，说话人姓名
    lines: string       # 必填，台词内容
    action: string      # 选填，说话时的伴随动作
```

---

## 字段详细说明

### scene（场景）
描述当前场景的时空背景，是整个剧本段落的定位锚点。

| 子字段 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| location | string | 是 | 场景发生的地点 | "青石岭祖祠后院" |
| time | string | 是 | 场景时间描述 | "深秋黄昏，斜阳熔金" |
| atmosphere | string | 否 | 场景整体氛围 | "肃穆中透着苍凉" |

### characters（人物）
描述该场景中出场的所有人物，是人物卡和人物关系图的数据来源。

| 子字段 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| name | string | 是 | 人物姓名 | "赵伯庸" |
| appearance | string | 否 | 人物外貌描述 | "鹤形松姿，眉骨高耸" |
| personality | string | 否 | 人物性格特征 | "沉毅如山，言少而重" |

### actions（动作）
描述场景中无台词的纯动作段落，是分镜脚本中动作镜头的直接来源。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| actions[] | string | 否 | 每条为一个独立动作描写 |

### dialogue（对话）
描述场景中的台词交流，包含说话人、台词内容和伴随动作。

| 子字段 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| speaker | string | 是 | 说话人姓名 | "朱砚舟" |
| lines | string | 是 | 台词内容 | "这树，当年咱们埋粮种时才及腰高。" |
| action | string | 否 | 说话时的伴随动作 | "指尖朝银杏主干偏左三寸处一点" |

## Schema 设计原因

### 一、为什么选择 YAML 而不是 JSON 或纯文本？

YAML 是本项目的核心格式选择，原因有三：

1. **可读性强**：YAML 的缩进结构让剧本作者（非技术用户）可以直接阅读和手动编辑，不需要处理 JSON 的括号、引号和逗号，大幅降低编辑门槛。
2. **结构清晰**：YAML 的层级缩进天然对应剧本的组织结构（场景→人物→动作→对话），视觉上直观。
3. **程序友好**：YAML 可以被 Python、JavaScript 等语言轻松解析，便于后端接口处理和前端渲染，同时也是 AI 模型容易生成的结构化格式。

相比纯文本，YAML 有明确的结构约束，保证了输出的一致性，使下游功能（人物卡、关系图、分镜）可以稳定解析。

---

### 二、为什么分成四个顶层字段？

这四个字段的划分不是随意的，而是经过设计，让同一份 YAML 文件能够驱动 Scriptcraft 的多个功能模块：

#### `scene` → 分镜脚本的环境基础
场景是影视和舞台创作的基本单位，镜头切换以场景为边界。`scene` 字段提供了分镜脚本生成所需的环境信息：`location` 决定镜头的背景布置，`time` 影响光线和色调，`atmosphere` 帮助 AI 判断镜头的情绪基调，从而生成合适的景别和运动方式。

#### `characters` → 人物卡 + 人物关系图的数据来源
`characters` 字段是 Scriptcraft 两个重要功能的数据基础：
- **人物卡**：系统解析 `name`、`appearance`、`personality` 三个字段，自动生成每个角色的人物卡片，作者可保存到人物卡库供后续参考。
- **人物关系图**：AI 读取所有场景中出现的人物姓名及其对话互动，分析角色之间的关系类型，生成可视化的人物关系网络图。

将人物信息集中在 `characters` 字段而不是散落在叙述中，是这两个功能能稳定运行的前提。

#### `actions` → 分镜脚本的动作镜头来源
`actions` 独立于 `dialogue`，对应电影剧本中无台词的纯动作段落。将动作和对话分离，使分镜 AI 能精确识别哪些镜头是动作镜头、哪些是对话镜头，从而分配合适的景别（动作镜头多用全景/中景）、运动方式（跟镜、推镜）和镜头时长。每条 `action` 可以对应分镜脚本中的一个独立镜头。

#### `dialogue` → 剧本核心内容 + 分镜台词来源
台词是剧本最核心的创作内容。`dialogue` 包含说话人、台词、伴随动作三要素，完整保留了对话的上下文信息，便于剧本评审、AI 修改，以及分镜脚本中台词字幕的生成。

---

### 三、为什么 dialogue 包含 action 字段而不是放在 actions 列表？

台词和动作往往同时发生，例如"（拍桌子）你凭什么！"。如果把伴随动作放在 `actions` 列表里，会失去与具体台词的绑定关系，导致分镜生成时无法判断这个动作属于哪句台词。将 `action` 嵌入 `dialogue` 保留了两者的绑定关系，符合好莱坞标准剧本格式（括号内动作指示）的惯例，也便于导演和演员直接使用。

---

### 四、为什么 actions 是列表而不是单个字符串？

一个场景中可能有多个连续动作，例如角色先开门、再走近、再拿起物品。列表结构支持任意数量的动作描写，不限制改编的丰富程度，同时每条 `action` 在分镜脚本生成时可以对应一个独立镜头，使分镜粒度更细腻。

---

### 五、为什么 appearance 和 personality 是可选字段？

同一人物在长篇小说改编中会在不同场景反复出现，只需在角色首次登场时描述外貌和性格，后续场景可以省略，避免重复冗余。这也符合实际剧本写作习惯：人物介绍只出现在角色第一次亮相时。设为可选字段既保持了灵活性，又不影响人物卡的生成（系统会合并同名角色的所有描述）。

---

### 六、Schema 与 Scriptcraft 功能链路的对应关系

Scriptcraft 构建了一条完整的创作链路：
小说原文
↓ AI 改编
YAML 剧本（本 Schema）
├── characters → 人物卡自动生成
├── characters → 人物关系图生成
├── scene + actions + dialogue → 分镜脚本生成
└── 全部字段 → 剧本评审与评分

正是因为 Schema 的结构清晰、字段职责单一，Scriptcraft 才能用同一份 YAML 文件驱动四个功能模块，实现「一次生成，多处复用」的设计目标。如果 Schema 设计不统一，例如人物信息散落在叙述中，或者动作和对话混在一起，这些下游功能就无法稳定解析，整个工具链就会断裂。
